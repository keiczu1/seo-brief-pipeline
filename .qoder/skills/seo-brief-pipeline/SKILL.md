# SEO Brief Pipeline

Универсальный pipeline: от поискового запроса до готовой HTML-статьи за 4 этапа.

## Когда использовать

- Пользователь даёт поисковый запрос и хочет получить production-ready бриф + HTML-статью
- Пользователь просит провести SEO-анализ запроса
- Пользователь просит написать статью на основе брифа
- Slash command: `/seo-brief-pipeline`

## Входные данные

Минимум:
- **Запрос** (query): поисковая фраза
- **Язык/регион**: ru-RU по умолчанию

Опционально:
- SERP XML (если есть в кэше проекта)
- Путь к `analysis-engine.mjs` (если доступен в проекте)
- Архетип страницы (если пользователь знает заранее)
- Ограничения (не покрывать X, обязательно включить Y)

## Pipeline: 4 этапа

```
Запрос
  │
  ▼
┌─────────────────────────────────────────────┐
│ Stage 1: Rule-Based Analysis                │
│ analysis-engine.mjs ИЛИ offline-режим       │
│ Выход: archetype, outline, entity map,      │
│        YMYL flag, source plan, gaps         │
└───────────────────┬─────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│ Stage 2: LLM Synthesis                      │
│ Заполнение 15-секционного брифа             │
│ Выход: полный ТЗ-бриф (markdown)            │
└───────────────────┬─────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│ Stage 3: Fact-Check + Score                 │
│ Проверка фактов через WebSearch + rubric    │
│ Выход: верифицированный бриф (score ≥ 32)   │
└───────────────────┬─────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│ Stage 4: HTML Article Generation            │
│ Генерация статьи по брифу                   │
│ Выход: HTML-файл со schema.org markup       │
└─────────────────────────────────────────────┘
```

## Stage 1: Rule-Based Analysis

### Если доступен `analysis-engine.mjs`

Запусти pipeline:

```powershell
npm.cmd run analyze:query -- --id <queryId> --query "<query>" --locale ru-RU
```

Прочитай результат: `data/analysis/full-analysis-output.json`

Извлеки:
- `archetype` — тип страницы (how_to_guide, comparison_page, etc.)
- `isYMYL` — YMYL-флаг
- `outline` — предлагаемая структура
- `entityMap` — сущности и атрибуты
- `sourcePlan` — требования к источникам
- `gaps` — пробелы в покрытии
- `profile` — профиль страницы

### Если `analysis-engine.mjs` недоступен (offline-режим)

Выполни rule-based анализ вручную:

1. **Определи архетип** по ключевым словам в запросе:
   - `как`, `способы`, `порядок`, `инструкция` → `how_to_guide`
   - `что такое`, `определение`, `значение` → `article_guide_definition`
   - `сравнение`, `vs`, `или`, `что лучше` → `comparison_page`
   - `калькулятор`, `конвертер`, `рассчитать` → `hybrid_tool_guide`
   - `топ`, `рейтинг`, `лучшие` → `commercial_comparison_page`
   - Медицинские/юридические/финансовые термины → добавить YMYL-флаг

2. **Смешанные архетипы.** Если запрос неоднозначен (например, «как выбрать и что лучше»):
   - Определи **доминантный intent** — что пользователь хочет сделать в первую очередь
   - Выбери архетип по доминанте + добавь элементы второго (FAQ, таблицы сравнения)
   - Запиши: `archetype: how_to_guide (с элементами comparison_page)`

3. **Определи YMYL** — запрос касается здоровья, права, финансов, безопасности?

4. **Проверь parsed-matrix** (если есть в проекте): файл `parser-output/parsed-matrix.json` содержит данные о SERP — используй их для enrichment.

5. **Построй базовый entity map** — основные сущности, их атрибуты, действия пользователя.

6. **Предложи outline** — 6–10 H2 секций, подходящих архетипу.

Запиши результат как `stage1_output` в начале брифа (внутренний блок, не для публикации).

**Evidence mode**: если SERP не парсился, пометь `evidence_mode: knowledge_draft`.

## Stage 2: LLM Synthesis

Это ключевой этап. Заполни **все 15 секций** брифа по шаблону в `references/brief-template.md`.

Подробные инструкции для каждой секции — в `references/stage2-instructions.md`.

### Краткий чек-лист секций

| # | Секция | Источник данных | WebSearch |
|---|--------|----------------|----------|
| 1 | Короткое решение | Stage 1 + анализ intent | Нет (заполнить последней) |
| 2 | Паспорт страницы | Stage 1 | Нет |
| 3 | Intent и пользовательская задача | Анализ запроса | Нет |
| 4 | Кластер запросов | Анализ intent + SERP | Опционально |
| 5 | Обязательные факты и SEO-триплеты | Entity map + предметная область | Нет (сбор, верификация в Stage 3) |
| 6 | Конкурентная картина | SERP (если есть) | Опционально |
| 7 | Стратегия нашей страницы | Секции 1–6 | Нет |
| 8 | Title, H1, description | Entity + intent + SERP | Нет |
| 9 | Подробный контент-план | Outline + секции 5–7 | Нет |
| 10 | UX и визуальные блоки | Архетип + intent | Нет |
| 11 | Источники и экспертная проверка | Секция 5 + YMYL-анализ | Нет (сбор, верификация в Stage 3) |
| 12 | Внутренние ссылки | Структура сайта (если есть) | Нет |
| 13 | Schema и технические требования | Архетип + контент-план | Нет |
| 14 | Редакционные правила | YMYL + стиль | Нет |
| 15 | Чек-лист приёмки | Все секции | Нет |

### Критические правила Stage 2

1. **Факты — собирай, но не верифицируй.** Stage 2 = сбор данных с указанием источников. Верификация — в Stage 3. Если цифра неизвестна, пиши `TBD — проверить в Stage 3` вместо выдуманного числа.
2. **SEO-триплеты с типами.** Каждый триплет должен иметь тип: `baseline` (обязательный), `enrichment` (улучшение), `risk` (не переборщить), `excluded` (не включать).
3. **Кластер из 4 групп.** Core (обязательно), Supporting (в секциях), Adjacent (ссылка/упоминание), Excluded (не покрывать).
4. **Контент-план с задачами.** Каждый H2 должен иметь задачу: что пользователь должен понять/сделать после секции.
5. **Связь триплет → секция.** Каждый baseline-триплет должен быть раскрыт в конкретной секции.

### Порядок заполнения секций

Секция 1 «Короткое решение» заполняется **последней**, отдельным проходом после секций 2–9. Заполни сначала 2–9, потом вернись и напиши секцию 1 как итоговый executive summary.

## Stage 3: Fact-Check + Score

> **GATE: переход к Stage 4 запрещён, пока Stage 3 не пройден.**
> После завершения Stage 3 выведи явный блок:
> ```
> ═══ STAGE 3 COMPLETE ═══
> Score: XX/40
> Fact-check: пройдена | частичная (N фактов unver)
> Переход к Stage 4: разрешён / запрещён (причина)
> ════════════════════════
> ```
> **Попроси подтверждение пользователя** перед генерацией HTML.

### 3.1. Проверка фактов

**Обязательно** через WebSearch проверь:

1. **Числовые данные**: госпошлины, сроки, штрафы, лимиты, проценты
2. **Номера статей**: ссылки на законы, стандарты, регламенты
3. **Актуальность**: данные не устарели? Были ли изменения в текущем году?
4. **URL источников**: ссылки ведут на реальные страницы?
5. **Противоречивые данные**: если WebSearch даёт разные числа — выбери наиболее свежий официальный источник и укажи альтернативу

**Протокол источников:**
- Приоритет: официальный сайт ведомства > КонсультантПлюс/Гарант > профильные СМИ
- Если 0 результатов: пометь факт как `unverified` и предупреди пользователя
- Если данные противоречат друг другу: приведи оба с указанием даты и источника

Если WebSearch находит расхождение с брифом — **обнови бриф** и отметь в секции 11 дату проверки.

### 3.2. Scoring по рубрике

Оцени бриф по шкале из 20 критериев (0-2 балла каждый, максимум 40):

| # | Критерий | Что проверяем | Критерий fail |
|---|----------|---------------|---------------|
| 1 | Intent fit | Бриф закрывает intent пользователя | Outline specificity |
| 2 | Page type | Архетип определён корректно | SERP discipline |
| 3 | First screen | Пользователь получает ответ за 5 сек | Source plan (для YMYL) |
| 4 | Modules | UX-блоки уместны и специфицированы | Entity map presence |
| 5 | Executive synthesis | Секция 1 ясна и компактна | Query cluster presence |
| 6 | Differentiating value | Есть чёткое преимущество | Implementation contract |
| 7 | Fact accuracy | Все числа проверены | — |
| 8 | Source quality | Источники официальные, URL рабочие | — |
| 9 | Triplet coverage | Все baseline раскрыты в секциях | — |
| 10 | Cluster completeness | Core-запросы покрыты | — |
| 11 | Competitor signals | Таблица сигнал→решение заполнена | — |
| 12 | FAQ quality | Вопросы из реальных страхов | — |
| 13 | Schema match | Schema описывает видимый контент | — |
| 14 | Editorial compliance | Правила слоя A/B соблюдены | — |
| 15 | No fabrication | Нет выдуманных URL/фактов | — |
| 16 | YMYL safeguard | Дисклеймер, дата, источники (если YMYL) | — |
| 17 | Excluded boundary | Excluded-запросы не раскрыты | — |
| 18 | Cannibalization | Проверена в паспорте | — |
| 19 | Evidence mode | Указан и соответствует реальности | — |
| 20 | Checklist ready | Секция 15 заполнена, все пункты ✓ | — |

**Пороги:**
- 38–40: production-ready
- 32–37: usable after revision (укажи, что доработать)
- < 32: redo Stage 2 (вернись и дополни слабые секции)

**Self-critique pass:** перед выставлением score — пройди по каждому критерию и найди слабое место. Если не можешь найти — спроси себя: «Что бы раскритиковал конкурент?»

### 3.3. Benchmark comparison (если доступен)

Если в проекте есть `benchmark/golden-queries.json` — сравни score с эталонным значением. Если score ниже эталона — укажи дельту и конкретные секции для улучшения.

## Stage 4: HTML Article Generation

Подробные правила — в `references/article-html-guide.md`.

### Краткие правила

1. **Семантический HTML5**: `<article>`, `<nav>`, `<section>`, `<details>`, `<time>`
2. **Schema.org JSON-LD** в `<head>`: `Article` + типовой schema (HowTo, FAQPage, etc.)
3. **Responsive CSS**: max-width 780px, media query ≤600px
5. **Структура**: breadcrumbs → sidebar TOC → meta → h1 → summary card → sections → FAQ → disclaimer
5. **Ссылки на источники**: `rel="noopener"`, `target="_blank"`
6. **Без выдуманных URL**: если не уверен в точном URL — используй base-URL документа

### Обязательные блоки для YMYL

- Дисклеймер: «Информация носит общий характер, не является юридической/медицинской консультацией»
- Дата актуальности для числовых данных
- Ссылки на официальные источники
- Образец документа (если тема предполагает составление документа)

Путь: `<project>/output/` (создать, если нет).

## Hard Rules

1. **Не выдумывай факты.** Числа, суммы, сроки, статьи законов — только после проверки. В Stage 2 пиши `TBD` если не уверен.
2. **Не делай одну структуру для всех запросов.** Архетип определяет outline.
3. **YMYL = обязательный дисклеймер + проверка источников.**
4. **Stage 3 нельзя пропустить.** Выведи GATE-блок и получи подтверждение перед Stage 4.
5. **HTML ≠ бриф.** Бриф = ТЗ для копирайтера. HTML = финальная статья.
6. **Evidence mode всегда виден.** `knowledge_draft` ≠ `parsed_html`.
7. **SEO-триплеты с типами.** Без типизации — это просто список фактов.

## Пример использования

Пользователь: `/seo-brief-pipeline "ипотечный калькулятор"`

Pipeline:
1. Stage 1: архетип = `hybrid_tool_guide`, YMYL = финансовый
2. Stage 2: 15 секций брифа (калькулятор, формула, примеры, FAQ)
3. Stage 3: проверка ставок ЦБ, лимитов, программ → обновление цифр
4. Stage 4: HTML с калькулятором, таблицами, HowTo schema

Пользователь: `/seo-brief-pipeline "как развестись без согласия жены"`

Pipeline:
1. Stage 1: архетип = `how_to_guide`, YMYL = юридический
2. Stage 2: 15 секций брифа (суд, документы, сроки, дети, FAQ)
3. Stage 3: проверка госпошлин, статей СК/ГПК → обновление сумм
4. Stage 4: HTML со шагами, таблицами, FAQPage schema

## Fallback-сценарии

| Ситуация | Действие |
|---|---|
| Нет `analysis-engine.mjs` | Stage 1 в offline-режиме, пометь `evidence_mode: knowledge_draft` |
| Нет SERP данных | Секция 6 (конкуренты) = `SERP-недоступен`, используй WebSearch |
| Нет данных о сайте | Секция 12 (внутренние ссылки) = placeholder |
| WebSearch недоступен | Предупреди пользователя, пометь факты как `unverified` |
| Score < 32 | Вернись к Stage 2, дополни слабые секции |
| Факты в Stage 2 неизвестны | Пиши `TBD — проверить в Stage 3`, не выдумывай |

---

## Протокол ошибок

| Ситуация | Действие |
|---|---|
| WebSearch вернул 0 результатов | Пометь факт `unverified`, предупреди пользователя, используй base-URL если доступен |
| WebSearch вернул противоречивые данные | Приведи оба значения с датой и источником, выбери свежий официальный |
| SERP XML повреждён | Попробуй reparse; если не вышло — fallback на `evidence_mode: knowledge_draft` |
| Stage 2 занял >80% context | Деградируй: сократи секции 6 и 12, пометь `truncated` |
| Пользователь отклонил GATE | Вернись к указанной секции, доработай, пройди Stage 3 заново |
| Benchmark score > нашего на 6+ | Пересмотри секции с наименьшим score, дополни |

## Связь с другими skill

Этот skill оркестрирует pipeline. Он использует:

- `references/brief-template.md` — шаблон 15-секционного брифа (что заполнять)
- `references/stage2-instructions.md` — инструкции LLM-синтеза (как заполнять)
- `references/article-html-guide.md` — правила HTML-генерации
- `references/editorial-rules-legal.md` — редакторские правила для юридической ниши
- `references/editorial-rules-general.md` — редакторские правила для общей ниши

**Выбор редакторских правил по нише:**

| Ниша | Файл |
|---|---|
| Юридическая (право, законы, суды) | `editorial-rules-legal.md` |
| Общая (how-to, обзоры, инструкции) | `editorial-rules-general.md` |
| Медицинская, финансовая | Пока не созданы — используй `general` + добавь YMYL-правила из `legal` |

Если в проекте есть `skills/seo-query-brief/` (Stage 1 pipeline), этот skill использует его вывод как вход Stage 2. Если нет — Stage 1 выполняется вручную.
