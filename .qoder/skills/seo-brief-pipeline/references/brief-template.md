# Шаблон ТЗ-брифа (15 секций)

Заполняй каждую секцию по порядку. Формат: markdown. Имя файла: `brief-<slug>.md`.

## Адаптивность

Не все 15 секций обязательны для каждого запроса. Объём зависит от архетипа и evidence mode.

| Секция | Обязательна | Опциональна / упрощаема |
|---|---|---|
| 0. Мета | Всегда | — |
| 1. Короткое решение | Всегда (заполнить последней) | — |
| 2. Паспорт | Всегда | — |
| 3. Intent | Всегда | — |
| 4. Кластер | Всегда | Для простых запросов — только Core + Excluded |
| 5. Факты + триплеты | Всегда | Минимум 5 триплетов (вместо 8–12) для knowledge_draft |
| 6. Конкуренты | Всегда | Для knowledge_draft — 1–2 конкурента, не 3–5 |
| 7. Стратегия | Всегда | — |
| 8. Title/H1/Desc | Всегда | — |
| 9. Контент-план | Всегда | Минимум 4 H2 (вместо 6–10) для простых запросов |
| 10. UX-блоки | Всегда | Минимум: summary-card + FAQ |
| 11. Источники | Всегда | Для knowledge_draft — без проверки URL |
| 12. Внутренние ссылки | Опциональна | Если нет данных о структуре сайта |
| 13. Schema | Всегда | — |
| 14. Редакторские правила | Всегда | — |
| 15. Чек-лист | Всегда | — |

**Минимальный бриф** (knowledge_draft + простой запрос): секции 0–5, 7–9, 11, 13–15 = 11 секций.

## 1. Короткое решение

За 30 секунд объясняет суть. Заполни после секций 2–9.

```
Делаем: <тип страницы>
Почему: <логика intent и SERP>
Для кого: <аудитория>
Главная задача: <что пользователь должен решить>
Главное отличие: <чем будем лучше среднего конкурента>
Главный риск: <что может испортить страницу>
```

## 3. Intent и пользовательская задача

Ответь на вопросы:
- Что пользователь хочет получить **сразу**?
- Какую проблему он решает?
- Какой результат он считает успешным?
- Что он уже, вероятно, знает?
- Какие **страхи, сомнения, ограничения** у него есть?
- Какой intent: информационный / коммерческий / транзакционный / смешанный?
- Какие intent **нельзя** смешивать на этой странице?

---

## 4. Кластер запросов и границы страницы

### 4 группы:

**Core** (обязательно покрыть на странице):
- <запрос 1>
- <запрос 2>

**Supporting** (раскрыть в секциях):
- <запрос 1> → секция H2: <название>
- <запрос 2> → секция H2: <название>

**Adjacent** (упомянуть или дать внутреннюю ссылку):
- <запрос> → куда ссылаться

**Excluded** (не покрывать на этой странице):
- <запрос> — почему нет

## 5. Обязательные факты и SEO-триплеты

### 5.1. Таблица обязательных фактов

| Факт | Почему обязателен | Где раскрыть | Источник | Статус |
|---|---|---|---|---|
| | | | | collected / TBD / verified |

> Статус `TBD` = факт неизвестен, будет проверен в Stage 3.

### 5.2. SEO-триплеты

Формат: `сущность → отношение → значение`

Типы:
- `baseline` — обязательно, без этого страница неполная
- `enrichment` — делает страницу лучше конкурентов
- `risk` — можно включить, но осторожно
- `excluded` — не включать

| Сущность | Отношение | Значение | Тип | Где использовать | Проверка |
|---|---|---|---|---|---|
| | | | | | |

Минимум: 8–12 триплетов.

## 6. Конкурентная картина

### Если SERP доступен:

Таблица топ-5 конкурентов с колонками: title, intent, сильные стороны, слабые стороны, сигнал.

### Если SERP недоступен (WebSearch):

Найди 3–5 статей по запросу, проанализируй структуру.

### Сигнал → Решение:

| Сигнал у конкурентов | Значение для нас | Решение | Где в ТЗ |
|---|---|---|---|
| | | | |

## 7. Стратегия нашей страницы

Заполни после секций 1–6:

- **Baseline** (минимальный уровень): что страница обязана закрыть
- **Advantage** (наше преимущество): чем мы лучше
- **Format advantage** (форматное преимущество): таблицы, схемы, калькуляторы, FAQ

## 9. Подробный контент-план

Для каждого H2:

```

Вопрос пользователя: <какой вопрос закрывает>
Что написать: <конкретика для копирайтера>

```

Минимум: 6–10 секций H2. Каждая секция = конкретная задача, не просто тема.

## 10. UX и визуальная структура

### Первый экран:
- короткий ответ
- карточка ключевых параметров
- сценарная развилка (если применимо)

### Навигация:
- sticky table of contents
- breadcrumbs

### Обязательные блоки:
- таблицы (для сравнения)
- callout-блоки (⚠️ warning, 💡 tip)
- timeline (для сроков/этапов)
- чек-лист (для подготовки)
- FAQ accordion (для частых вопросов)
- образец документа (для тем с составлением документов)
- судебная практика (для конфликтных юридических тем)
- таблица рисков (для сложных споров)

### Сценарные развилки:
Если запрос допускает несколько путей — опиши развилку и точки ветвления.

## 13. Schema и технические SEO-требования

### Обязательные schema:

```json
{
  "@type": "Article",
  ...
}
```

Дополнительные schema по архетипу:
- `how_to_guide` → `HowTo`
- FAQ в контенте → `FAQPage`
- `comparison_page` → `ItemList`
- `hybrid_tool_guide` → `SoftwareApplication`
- `local_service_page` → `LocalBusiness`

### Технические требования:
- Title: 55–60 символов
- Description: 145–155 символов
- H1: один на странице
- H2/H3: иерархия без пропусков
- Alt-тексты для изображений

---

## 14. Редакционные правила

### Стиль:
- Обращение: на «вы» / на «ты» (по контексту)
- Тон: экспертный, но доступный
- Длина предложений: до 25 слов
- Абзацы: 2–4 предложения

### YMYL-правила (если применимо):
- Дисклеймер: «Информация носит общий характер и не является <юридической/медицинской/финансовой> консультацией»
- Разделять общие правила и ситуации, зависящие от обстоятельств
- Не давать универсальных обещаний

### Запрещено:
- Выдуманные URL
- Устаревшие данные без даты проверки
- Обещания ранжирования

## 15. Чек-лист приёмки

```
- [ ] Title/H1/Description соответствуют секции 8
- [ ] Каждый baseline-триплет из секции 5 раскрыт в контент-плане
- [ ] Content-план покрывает все core-запросы из кластера
- [ ] FAQ закрывает основные сомнения пользователя (секция 3)
- [ ] UX-блоки из секции 10 специфицированы
- [ ] YMYL-дисклеймер присутствует (если применимо)
- [ ] Источники проверены (секция 11)
- [ ] Schema соответствует видимому контенту
- [ ] Excluded-запросы из кластера НЕ раскрыты на странице
- [ ] Факты с номерами статей/суммами проверены через WebSearch
- [ ] Дата актуальности указана для чувствительных данных
- [ ] Образец документа включён (если тема предполагает составление документа: иск, заявление, жалоба, претензия, договор)
- [ ] Судебная практика включена (для конфликтных юридических тем, 2-3 анонимных примера)
- [ ] Таблица рисков включена (для сложных споров с несколькими рисками)
- [ ] Цитаты из законов оформлены курсивом (для юридической ниши)
- [ ] Термины при первом упоминании выделены bold + пояснение
```
