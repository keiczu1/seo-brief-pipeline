# Редакторский стандарт: юридический контент + SEO-pipeline

Объединённый документ. Слой A — редакционные правила (контент). Слой B — SEO/технические правила (pipeline). Оба слоя обязательны.

---

## СЛОЙ A: РЕДАКЦИОННЫЕ ПРАВИЛА (КОНТЕНТ)

### A1. Целевая аудитория и тон

**Кто наш читатель:**
Юридически неподготовленный человек, столкнувшийся с правовой проблемой. Не знает законов, не понимает терминологию, боится ошибок и судов. Ему нужна не теория, а практическая помощь: что делать, как делать, сколько стоит, какие риски.

**Тон (4 характеристики):**
- **Уважительный** — не свысока, термины объясняем
- **Эмпатичный** — понимаем, что читатель может быть в стрессе
- **Честный** — не обещаем лёгких решений
- **Практичный** — даём инструменты, а не просто информацию

### A2. Интенты — закрываем все

| Интент | Вопрос пользователя | Как закрываем |
|---|---|---|
| Информационный | «Что это?», «Это про меня?» | Краткое введение, суть, перечень ситуаций |
| Навигационный | «Какой вариант мне подходит?» | Сценарии: «если А — делайте так, если Б — иначе» |
| Практический | «Что делать прямо сейчас?» | Пошаговый алгоритм с нумерацией |
| Инструментальный | «Где взять бланк?», «Что собрать?» | Шаблоны, чек-листы, перечни документов |
| Транзакционный | «Мне нужен юрист?» | Честно: когда можно самому, когда нужен специалист |

### A3. Структура — от проблемы к решению

Логика движения (рекомендуемый порядок, можно адаптировать):

1. **Суть вопроса** — кратко, главное правило, для кого актуально
2. **Жизненные ситуации** — примеры, чтобы читатель узнал себя
3. **Варианты решения** — обзор путей, помощь в выборе
4. **Пошаговый алгоритм** — ядро статьи, чёткая инструкция
5. **Что нужно** — документы, сроки, стоимость
6. **Что может пойти не так** — риски, ограничения, частые ошибки
7. **Особые случаи** — нюансы, выбивающиеся из общего правила
8. **Резюме** — 3–5 пунктов глагол + суть

Адаптация по типу статьи:
- **Процедурная** → акцент на шагах, документах, сроках
- **Конфликтная** → акцент на досудебном порядке, судебной практике, рисках
- **О правах** → акцент на правах, алгоритме защиты, куда жаловаться
- **О документе** → акцент на структуре, пошаговом заполнении, образце

### A4. Язык и стиль

- **Второе лицо** («вы», «ваш») — обращение к читателю
- **Короткие предложения** — не более 15–20 слов
- **Активные глаголы** — «возьмите», «подайте», «соберите», «направьте»
- **Без пассива** — не «иск должен быть подан», а «вам нужно подать иск»
- **Объяснение терминов** — при первом упоминании: «досудебная претензия — это официальное письмо…»
- **Честность** — «в некоторых случаях суд может отказать, если…»
- **Абзацы** — 2–4 предложения

### A5. Работа с сущностями

- При первом упоминании **выделяйте** жирным и давайте краткое пояснение
- Связывайте сущности между собой (причинно-следственные связи)
- Используйте синонимы: «изменить договор» / «пересмотреть условия» / «внести поправки»

### A6. Ссылки на законодательство

- Каждое ключевое утверждение → **ссылка на конкретную статью** + пояснение простыми словами
- Формат: *«По статье 22 СК РФ (ссылка) суд может назначить примирительный срок до 3 месяцев. Это значит, что даже если вы уверены в решении, процесс может затянуться».*
- Источники: КонсультантПлюс, Гарант, Официальный портал правовой информации, Судебные акты РФ
- **Ссылки ведут на конкретную статью, а не на общий документ.** Пример: `consultant.ru/document/cons_doc_LAW_8982/abc123/` (ст. 22 СК РФ), а НЕ `consultant.ru/document/cons_doc_LAW_8982/` (весь СК РФ). Пользователь должен сразу увидеть нужную норму, а не искать её в длинном документе.
- **В конце статьи** — блок **«Правовая база»**: все использованные нормы с гиперссылками и краткими пояснениями

### A7. Практические инструменты

| Инструмент | Когда нужен |
|---|---|
| Шаблоны документов | Тема предполагает составление обращения (иск, претензия, заявление) |
| Чек-листы | Многошаговая процедура (можно распечатать и отмечать) |
| Таблицы | Сравнение вариантов, сроков, рисков |
| Примеры из жизни | Всегда (вымышленные, но реалистичные) |

### A8. Образцы документов

Если статья предполагает составление документа — образец **обязателен**:
1. **В тексте** — полный шаблон с полями `[Ваше Ф.И.О.]`, `[Дата]`, `[Сумма]`
2. **Для скачивания** — кнопка на .docx (основной) и .pdf (дополнительный)
3. Дисклеймер: *«Образец носит информационный характер. Рекомендуем проконсультироваться с юристом».*

### A9. Судебная практика

- **Процедурные темы** — не нужна
- **Конфликтные темы** — обязательна: 2–3 анонимных примера
- Формат: *«ситуация → что решил суд → почему»*
- Анонимность: «в одном из дел», «Верховный суд указал, что…»

### A10. Риски и ограничения

- Простые процедуры → краткий блок «на что обратить внимание»
- Сложные споры → таблица «Риск → Как избежать»
- Если риск нельзя избежать — пишем прямо

### A11. FAQ

- Формулируем вопросы так, как задают реальные люди: «А что, если…?», «Сколько времени?», «Нужен ли адвокат?»
- Каждый ответ — со ссылкой на источник
- **НЕ дублируй контент статьи.** FAQ дополняет, а не повторяет. Если факт уже раскрыт в секции H2 — добавь нюанс, практический совет или edge-case. Проверка: если ответ можно скопировать из body-секции — переформулируй вопрос или замени его

### A12. Резюме — обязательно

3–5 пунктов. Формат: глагол + суть.
```
1. Определите подсудность: мировой суд (нет спора о детях) или районный.
2. Соберите документы и оплатите госпошлину 5 000 ₽.
3. Подайте иск и дождитесь заседания через ~1 месяц.
4. Если суд назначит примирительный срок — заявите, что примирение невозможно.
5. После решения — зарегистрируйте развод в ЗАГС (ещё 5 000 ₽).
```

### A13. Актуальность

- **Год в заголовке**: «Как развестись без согласия жены в 2026 году»
- Дата последнего обновления видна
- Изменения в законе за текущий год — отдельным блоком

### A14. Визуализация

- H2/H3 для структуры
- Маркированные списки для перечислений
- Нумерованные для пошаговых инструкций
- **Жирный** для ключевых терминов и выводов
- *Курсив* для цитат из законов
- Блоки «Важно!» для критических моментов
- Таблицы для сравнения
- **Схемы-развилки** для ветвлений:
  **НЕ использовать моноширинные ASCII-деревья** (`├──`, `└──` в ` ``` `). Они выглядят чужеродно и плохо читаются.
  Вместо них — **две кликабельные карточки** (`.choice-cards`): вопрос-подзаголовок → две карточки рядом (иконка + заголовок + описание + ссылка). Hover: accent border + тень. На мобильных — стек в колонку. См. `article-html-guide.md`.

## СЛОЙ B: SEO/ТЕХНИЧЕСКИЕ ПРАВИЛА (PIPELINE)

### B1. Архетип страницы

Архетип определяет структуру. Определяется из запроса:
- `как`, `порядок`, `инструкция` → `how_to_guide`
- `что такое`, `определение` → `article_guide_definition`
- `сравнение`, `или`, `что лучше` → `comparison_page`
- `калькулятор`, `рассчитать` → `hybrid_tool_guide`
- Юридические/медицинские/финансовые термины → уровень YMYL-риска (нет / средний / высокий)

### B2. SEO-триплеты с типами

Формат: `сущность → отношение → значение`

| Тип | Значение |
|---|---|
| `baseline` | Обязательно, без этого страница неполная |
| `enrichment` | Делает лучше конкурентов |
| `risk` | Можно, но осторожно |
| `excluded` | Не включать на этой странице |

Минимум 8–12 триплетов. Каждый baseline → конкретная секция H2.

### B3. Кластер запросов

| Группа | Роль |
|---|---|
| Core | Обязательно покрыть |
| Supporting | Раскрыть в секциях H2 |
| Adjacent | Упомянуть + внутренняя ссылка |
| Excluded | Не покрывать (отдельная страница) |

### B4. Конкурентная картина

Таблица «Сигнал → Значение → Решение → Где в ТЗ». Переводим наблюдение в действие.

### B5. Schema.org

| Архетип | Schema |
|---|---|
| Всегда | `Article` |
| `how_to_guide` | + `HowTo` |
| Есть FAQ | + `FAQPage` |
| `comparison_page` | + `ItemList` |

Schema описывает **только видимый** контент.

### B6. Fact-check (Stage 3)

Обязательно через WebSearch:
- Числовые данные (госпошлины, сроки, лимиты)
- Номера статей законов
- Актуальность (изменения в текущем году)
- URL источников

### B7. UX-блоки

| Блок | Когда |
|---|---|
| Summary card | Есть ключевые параметры (стоимость, срок) |
| Сценарная развилка | Intent зависит от ситуации |
| Sticky TOC | Секций > 5 |
| Callout ⚠️ / 💡 | Предупреждения и рекомендации |
| Timeline | Сроки и этапы |
| Чек-лист | Подготовка к действию |
| FAQ accordion | Частые вопросы (5+) |
| Образец документа | Тема = составление документа (иск, заявление, жалоба) |
| Судебная практика | Тема = конфликтный спор (юридич.) |
| Таблица рисков | Тема = сложный спор с рисками |

### B8. Cannibalization

В паспорте указываем существующие страницы, с которыми возможен конфликт.

### B9. Evidence mode

| Режим | Значение |
|---|---|
| `parsed_html` | SERP реально парсился |
| `title_snippet_only` | Только titles/snippets |
| `knowledge_draft` | Офлайн-анализ |

### B10. Scoring

22 критерия × 0-2 = 44 максимум. Порог production-ready: ≥ 38.

## АВТОРСКИЙ ЧЕК-ЛИСТ (A + B)

Перед сдачей статьи:

```
КОНТЕНТ (слой A):
- [ ] Заголовок содержит год (2026)
- [ ] Введение: для кого и зачем
- [ ] Тон: уважительный, эмпатичный, честный, практичный
- [ ] Предложения ≤ 20 слов, без пассива, активные глаголы
- [ ] Термины пояснены при первом упоминании
- [ ] Сущности выделены, связаны, с синонимами
- [ ] Каждое утверждение → ссылка на статью закона + пояснение
- [ ] Блок «Правовая база» в конце
- [ ] Шаблоны документов (если нужны) — в тексте + для скачивания
- [ ] Судебная практика (для конфликтных тем) — 2-3 примера
- [ ] Таблица рисков (для сложных тем)
- [ ] Резюме: 3-5 пунктов глагол + суть
- [ ] FAQ: вопросы как задают реальные люди; НЕ дублирует контент статьи
- [ ] Дисклеймер: информационный характер, не замена консультации
- [ ] Дата обновления видна
- [ ] Ссылки на законы — ведут на конкретную статью (URL с хешем), а не на общий документ
- [ ] Нет фабрикованных URL

SEO/ТЕХНИЧЕСКИЙ (слой B):
- [ ] Title (55-60 симв.) / H1 / Description (145-155 симв.)
- [ ] Schema.org: Article + типовой schema
- [ ] Каждый baseline-триплет раскрыт в секции
- [ ] Core-запросы кластера покрыты
- [ ] Excluded-запросы НЕ раскрыты
- [ ] Факты проверены через WebSearch
- [ ] Нет фабрикованных URL
- [ ] Responsive CSS (≤ 600px)
```

## ИТОГОВЫЙ ПРИНЦИП

> Не пишите по шаблону. Пишите по логике.
> Определите intent, архетип и сущности — и стройте страницу так, чтобы пользователь получил ответ и понял, что делать.
> Структура зависит от темы и архетипа — главное, чтобы она была логичной, полезной и понятной для юридически неподготовленного человека.
