# Редакторский стандарт: общий контент + SEO-pipeline

Объединённый документ для **не-YMYL** и **слабо-YMYL** ниш (how-to, обзоры, инструкции, сравнения). Для юридической ниши используй `editorial-rules-legal.md`. Для медицинской/финансовой — `general` + YMYL-правила из `legal`.

---

## СЛОЙ A: РЕДАКЦИОННЫЕ ПРАВИЛА (КОНТЕНТ)

### A1. Целевая аудитория и тон

**Кто наш читатель:**
Человек, который ищет решение конкретной задачи. Не эксперт, но мотивирован разобраться. Ему нужна не теория, а практический результат: что делать, как делать, сколько стоит, какие варианты.

**Тон (4 характеристики):**
- **Уважительный** — не свысока, термины объясняем
- **Практичный** — даём инструменты, а не просто информацию
- **Честный** — не преувеличиваем простоту, указываем ограничения
- **Дружелюбный** — но не фамильярный; нейтрально-позитивный стиль

### A2. Интенты — закрываем все

| Интент | Вопрос пользователя | Как закрываем |
|---|---|---|
| Информационный | «Что это?», «Зачем это нужно?» | Краткое введение, суть |
| Навигационный | «Какой вариант мне подходит?» | Сценарии, comparison blocks |
| Практический | «Что делать?» | Пошаговый алгоритм |
| Инструментальный | «Где взять?», «Что нужно?» | Шаблоны, чек-листы, перечни |
| Транзакционный | «Где купить?», «Сколько стоит?» | Таблицы цен, сравнение предложений |

### A3. Структура — от проблемы к решению

Логика движения:

1. **Суть вопроса** — кратко, для кого актуально
2. **Варианты решения** — обзор путей, помощь в выборе (choice-cards если есть развилка)
3. **Пошаговый алгоритм** — ядро статьи
4. **Что нужно** — инструменты, материалы, стоимость
5. **Что может пойти не так** — частые ошибки, ограничения
6. **Резюме** — 3–5 пунктов глагол + суть

Адаптация по типу:
- **How-to** → акцент на шагах, таймлайне
- **Обзор** → акцент на сравнении, таблицах
- **Определение** → акцент на сути, примерах использования
- **Сравнение** → акцент на таблицах, pros/cons

### A4. Язык и стиль

- **Второе лицо** («вы», «ваш») — обращение к читателю
- **Короткие предложения** — не более 20 слов
- **Активные глаголы** — «сделайте», «выберите», «подключите»
- **Без пассива** — не «должен быть установлен», а «установите»
- **Объяснение терминов** — при первом упоминании
- **Абзацы** — 2–4 предложения

### A5. Работа с сущностями

- При первом упоминании **выделяйте** жирным и давайте краткое пояснение
- Используйте синонимы для обогащения текста
- Связывайте сущности между собой

### A6. Источники и ссылки

- Ключевые утверждения → ссылка на источник
- Источники: официальные сайты, документация, авторитетные издания
- **Никогда не фабрикуют URL** — только verified или base-URL

### A7. Практические инструменты

| Инструмент | Когда нужен |
|---|---|
| Чек-листы | Многошаговая процедура |
| Таблицы | Сравнение вариантов, характеристик |
| Примеры | Конкретные use-cases |
| Шаблоны | Если читатель составляет документ/конфиг |

### A8. Визуализация

- H2/H3 для структуры
- Маркированные списки для перечислений
- Нумерованные для пошаговых инструкций
- **Жирный** для ключевых терминов и выводов
- Блоки callout для предупреждений (⚠️) и рекомендаций (💡)
- Таблицы для сравнения
- **Choice-cards** для ветвлений (НЕ ASCII-деревья)

### A9. Резюме — обязательно

3–5 пунктов. Формат: глагол + суть. В callout-блоке `.callout-summary` в конце статьи.

### A10. Актуальность

- **Год в заголовке** (если данные меняются): «Как настроить X в 2026 году»
- Дата обновления видна
- Числовые данные — с указанием даты актуальности

### A11. FAQ

- Вопросы как задают реальные люди
- Каждый ответ — самодостаточный (читатель может не читать остальное)
- **НЕ дублируй контент статьи.** FAQ дополняет, а не повторяет. Если факт уже раскрыт в секции H2 — добавь нюанс, практический совет или edge-case
- Для YMYL — со ссылкой на источник

## СЛОЙ B: SEO/ТЕХНИЧЕСКИЕ ПРАВИЛА (PIPELINE)

### B1. Архетип страницы

Архетип определяет структуру. Определяется из запроса:
- `как`, `порядок`, `инструкция` → `how_to_guide`
- `что такое`, `определение` → `article_guide_definition`
- `сравнение`, `или`, `что лучше` → `comparison_page`
- `калькулятор`, `рассчитать` → `hybrid_tool_guide`
- Юридические/медицинские/финансовые термины → уровень YMYL-риска (нет / средний / высокий)

### B2. SEO-триплеты с типами

Формат: `сущность → отношение → значение`

| Тип | Значение |
|---|---|
| `baseline` | Обязательно, без этого страница неполная |
| `enrichment` | Делает лучше конкурентов |
| `risk` | Можно, но осторожно |
| `excluded` | Не включать на этой странице |

Минимум 8–12 триплетов (5 для knowledge_draft). Каждый baseline → конкретная секция H2.

### B3. Кластер запросов

| Группа | Роль |
|---|---|
| Core | Обязательно покрыть |
| Supporting | Раскрыть в секциях H2 |
| Adjacent | Упомянуть + внутренняя ссылка |
| Excluded | Не покрывать (отдельная страница) |

### B4. Конкурентная картина

Таблица «Сигнал → Значение → Решение → Где в ТЗ». Переводим наблюдение в действие.

### B5. Schema.org

| Архетип | Schema |
|---|---|
| Всегда | `Article` |
| `how_to_guide` | + `HowTo` |
| Есть FAQ | + `FAQPage` |
| `comparison_page` | + `ItemList` |

Schema описывает **только видимый** контент.

### B6. Fact-check (Stage 3)

Обязательно через WebSearch:
- Числовые данные (цены, сроки, характеристики)
- Технические параметры, версии, даты релизов
- URL источников

### B7. UX-блоки

| Блок | Когда |
|---|---|
| Summary card | Есть ключевые параметры |
| Choice-cards | Intent зависит от ситуации |
| Sticky TOC | Секций > 5 |
| Callout ⚠️ / 💡 | Предупреждения и рекомендации |
| Timeline | Сроки и этапы |
| Чек-лист | Подготовка к действию |
| FAQ accordion | Частые вопросы (5+) |
| Образец документа | Тема = составление документа |
| Судебная практика | Тема = конфликтный спор (юридич. / YMYL) |
| Таблица рисков | Тема = сложный спор с рисками |
| Резюме | Всегда (3–5 пунктов) |

### B8. Cannibalization

В паспорте указываем существующие страницы, с которыми возможен конфликт.

### B9. Evidence mode

| Режим | Значение |
|---|---|
| `parsed_html` | SERP реально парсился |
| `title_snippet_only` | Только titles/snippets |
| `knowledge_draft` | Офлайн-анализ |

### B10. Scoring

22 критерия × 0-2 = 44 максимум. Порог production-ready: ≥ 38.

## АВТОРСКИЙ ЧЕК-ЛИСТ (A + B)

Перед сдачей статьи:

```
КОНТЕНТ (слой A):
- [ ] Заголовок содержит год (если актуально)
- [ ] Введение: для кого и зачем
- [ ] Тон: уважительный, практичный, честный, дружелюбный
- [ ] Предложения ≤ 20 слов, без пассива, активные глаголы
- [ ] Термины пояснены при первом упоминании
- [ ] Сущности выделены и связаны
- [ ] Ключевые утверждения → ссылки на источники
- [ ] Резюме: 3-5 пунктов глагол + суть
- [ ] FAQ: вопросы как задают реальные люди; НЕ дублирует контент статьи
- [ ] Дата обновления видна
- [ ] Нет фабрикованных URL

SEO/ТЕХНИЧЕСКИЙ (слой B):
- [ ] Title (55-60 симв.) / H1 / Description (145-155 симв.)
- [ ] Schema.org: Article + типовой schema
- [ ] Каждый baseline-триплет раскрыт в секции
- [ ] Core-запросы кластера покрыты
- [ ] Excluded-запросы НЕ раскрыты
- [ ] Факты проверены (Stage 3)
- [ ] Нет фабрикованных URL
- [ ] Responsive CSS (desktop + tablet + mobile)
- [ ] Резюме-блок присутствует
```

> Определите intent, архетип и сущности — и стройте страницу так, чтобы пользователь получил ответ и понял, что делать.
