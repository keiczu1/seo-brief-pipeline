# Stage 2: Инструкции LLM-синтеза

Этот документ описывает, **как** заполнять каждую секцию брифа. Это Stage 2 pipeline — LLM-синтез поверх rule-based анализа (Stage 1).

---

## Общий принцип

Stage 1 даёт ~40% брифа:
- Архетип страницы
- Базовый outline
- Entity map (сущности и атрибуты)
- YMYL-флаг
- Source plan (какие факты нужны)

Stage 2 заполняет оставшиеся ~60%:
- Кластер запросов (4 группы)
- SEO-триплеты с типами
- Конкурентная картина (сигнал → решение)
- Стратегия страницы
- Подробный контент-план с задачами
- UX-блоки и сценарные развилки
- FAQ вопросы и ответы
- Редакционные правила
- Чек-лист приёмки

---

## Секция 1: Короткое решение

**Когда заполнять:** **последним**, отдельным проходом после секций 2–9.

Это итоговый executive summary. Заполни сначала все остальные секции, затем вернись к секции 1.

**Как:**
- «Делаем» = архетип + формат (например, «юридический гайд с пошаговым порядком»)
- «Почему» = intent + почему эта структура подходит
- «Для кого» = конкретная аудитория (не «все», а «мужчина 30-55, принял решение о разводе»)
- «Главная задача» = что пользователь должен решить после прочтения
- «Главное отличие» = чем будем лучше топ-3 конкурентов
- «Главный риск» = что может испортить страницу

---

## Секция 2: Паспорт

**Когда:** сразу после Stage 1.

**Как:**
- YMYL: если запрос касается здоровья, права, финансов, безопасности → высокий
- Экспертная проверка: для YMYL = да, указать кто (юрист, врач, финансовый консультант)
- Cannibalization: подумай, какие другие страницы сайта могут перекрывать intent

---

## Секция 3: Intent

**Когда:** после паспорта.

**Как:**
- Не «о чём запрос», а «что человек пытается сделать»
- Перечисли конкретные **страхи и сомнения** пользователя — они станут FAQ
- Определи, какие intent **нельзя** смешивать (они уйдут в excluded-кластер)

**Пример хорошей формулировки:**
```
Пользователь не ищет определение. Он уже принял решение и ищет порядок действий.
Ему нужны: пошаговый план, список документов, сроки, стоимость, что делать с детьми.
Его страхи: «а вдруг суд откажет?», «что если жена будет тянуть?», «сколько это стоит?»
```

---

## Секция 4: Кластер запросов

**Когда:** после intent.

**Как:**

**Core** — запросы, которые эта страница ОБЯЗАНА закрыть:
- Перефразировки основного запроса
- Уточнения (с детьми, без детей, в другом городе)

**Supporting** — запросы, которые раскрываются в отдельных H2:
- Каждый supporting → конкретная секция в контент-плане

**Adjacent** — связанные темы для внутренних ссылок:
- НЕ раскрывать подробно на этой странице
- Только ссылка на отдельную страницу

**Excluded** — что НЕ покрывать:
- Противоположный intent (например, «как помириться»)
- Темы, требующие отдельной страницы (подробный раздел имущества)

---

## Секция 5: Обязательные факты и SEO-триплеты

**Когда:** после кластера.

### Обязательные факты

Это конкретные данные, без которых страница неполная:
- Числа (госпошлины, сроки, лимиты)
- Ссылки на законы/стандарты
- Конкретные процедуры

**ВАЖНО: Stage 2 = сбор фактов с указанием источников.** Верификация через WebSearch — в Stage 3.
Если цифра неизвестна — пиши `TBD — проверить в Stage 3`, не выдумывай.
Если известна — укажи значение + источник + дату, но не проверяй заново.

### SEO-триплеты

Формат: `сущность → отношение → значение`

**Типы (обязательно указать):**

| Тип | Значение | Пример |
|---|---|---|
| `baseline` | Без этого страница неполная | «госпошлина → составляет → 5 000 руб.» |
| `enrichment` | Делает лучше конкурентов | «несогласие → не может предотвратить → развод» |
| `risk` | Можно, но осторожно | «спор о детях → рассматривается → районным судом» |
| `excluded` | Не включать | «быстрый развод → невозможен → без суда» |

**Правила:**
- Минимум 8–12 триплетов
- Каждый baseline → конкретная секция в контент-плане
- Каждый risk → указать, где НЕ переборщить
- Каждый excluded → убедиться, что не раскрыт

---

## Секция 6: Конкурентная картина

**Когда:** после триплетов.

### Если SERP доступен:
Проанализируй топ-5 результатов. Для каждого: title, intent, сильные/слабые стороны.

### Если SERP недоступен:
WebSearch → найди 3–5 статей по запросу → проанализируй структуру.

**Примечание:** конкурентный анализ через WebSearch даёт приблизительную картину. Если результаты нерелевантны — пометь `competitor_data: weak` и переходи дальше.

### Сигнал → Решение:

Это ключевая таблица. Переведи наблюдение в действие:

| Сигнал | Значение | Решение | Где в ТЗ |
|---|---|---|---|
| Все конкуренты начинают с длинного введения | Пользователь уходит, не найдя ответ | Начать с короткого ответа | Первый экран |
| У конкурентов нет таблицы сроков | Пользователь не видит timeline | Сделать визуальный timeline | Секция «Сроки» |

---

## Секция 7: Стратегия

**Когда:** после конкурентной картины.

Заполни:
- **Baseline**: что страница ОБЯЗАНА закрыть (из core-кластера + baseline-триплетов)
- **Advantage**: чем мы лучше (из конкурентного анализа)
- **Format advantage**: какой формат даст преимущество (таблицы, timeline, FAQ accordion, сценарные развилки)

---

## Секция 8: Title, H1, Description

**Когда:** после стратегии.

**Title** (55–60 символов):
- Содержит основной запрос
- Указывает тип страницы (порядок, инструкция, обзор)
- Добавляет год если актуально

**H1**:
- Один на странице
- Может отличаться от title
- Полный, естественный заголовок

**Description** (145–155 символов):
- Описывает, что пользователь получит
- Содержит ключевые параметры (срок, стоимость, количество шагов)

---

## Секция 9: Контент-план

**Когда:** это центральный блок Stage 2.

Для КАЖДОГО H2 заполни:

```
### H2: <Название>

Задача секции: <что пользователь должен понять/сделать>
Вопрос пользователя: <какой конкретный вопрос закрывает>
Что написать: <детальная инструкция для копирайтера — не тема, а конкретика>
Обязательные факты: <из секции 5, с номерами статей/суммами>
Покрываемые запросы: <из кластера, core или supporting>
Формат блока: текст | таблица | нумерованный список | callout | timeline | чек-лист | document-template
Не писать: <чего избегать в этой секции>
```

**Критические правила:**
- Каждый H2 = задача, не просто тема
- «Что написать» = конкретные инструкции, не «рассказать о...»
- Факты привязаны к источникам
- Формат выбран осознанно (таблица для сравнения, timeline для сроков, callout для предупреждений)

### Проверка: нужен ли образец документа

**Обязательный чек после контент-плана.** Пройди по списку:

| Признак | Нужен образец? |
|---|---|
| Статья о подаче иска/заявления/жалобы/претензии | **Да** |
| Статья о составлении договора/документа | **Да** |
| Статья о регистрации/оформлении (без документа) | Нет, только перечень |
| How-to без документа (настройка, установка) | Нет |
| Информационная/обзорная статья | Нет |

Если **хотя бы один признак** указывает на «Да»:
1. Добавь H2 «Образец документа» в контент-план с форматом `document-template`
2. В секции 10 (UX) добавь блок «Образец документа» с описанием
3. В чек-листе (секция 15) отметь пункт об образце

**Если не уверен — включи.** Лучше дать лишний образец, чем упустить.

**FAQ секция:**
- Бери вопросы из страхов/сомнений пользователя (секция 3)
- Минимум 5–7 вопросов
- Каждый ответ содержит ссылку на источник

---

## Секция 10: UX и визуальные блоки

**Когда:** параллельно с контент-планом.

Определи:
- **Сценарные развилки**: если intent зависит от ситуации (есть дети / нет детей)
- **Sticky nav**: если секций > 5
- **Карточка параметров**: если есть ключевые числа (стоимость, срок)
- **Callout-блоки**: ⚠️ для предупреждений, 💡 для рекомендаций
- **Timeline**: для этапов/сроков
- **Чек-лист**: для подготовки
- **Образец документа**: если тема предполагает составление (иск, заявление, жалоба)
- **Судебная практика**: если тема конфликтная (спор, несогласие, раздел)
- **Таблица рисков**: если тема сложная с несколькими рисками

### Обязательный UX-чек

После определения блоков — пройди по таблице и отметь, какие блоки нужны:

| Блок | Условие | Эта тема? |
|---|---|---|
| Summary card | Есть ключевые параметры (цена, срок, сумма) | да / нет |
| Choice-cards | Intent зависит от ситуации | да / нет |
| Sidebar TOC | Секций > 5 | да / нет |
| Callout ⚠️ | Есть предупреждения | да / нет |
| Callout 💡 | Есть рекомендации | да / нет |
| Timeline | Есть этапы/сроки | да / нет |
| Чек-лист | Есть подготовка к действию | да / нет |
| FAQ accordion | Есть 5+ частых вопросов | да / нет |
| Резюме | Всегда | да |
| **Образец документа** | Тема = составление документа | да / нет |
| **Судебная практика** | Тема = конфликтный спор (юридич.) | да / нет |
| **Таблица рисков** | Тема = сложный спор с рисками | да / нет |
| **Цитаты курсивом** | Юридическая ниша (ссылки на законы) | да / нет |
| **Термины bold** | Есть термины для пояснения | да / нет |

Если отметил «да» — добавь блок в секцию 10 и в контент-план (секция 9).

---

## Секция 11: Источники

**Когда:** после контент-плана.

Для каждого факта из секции 5:
- Укажи конкретный URL (если известен)
- Статус: `source_collected` | `source_search_required`
- **Верификация URL — в Stage 3.** Здесь достаточно указать источник.

Для YMYL: добавь `reviewer_required: да/нет`.

---

## Секция 12: Внутренние ссылки

**Когда:** после источников.

Если структура сайта неизвестна:
- Укажи Adjacent-запросы из кластера как кандидаты
- Пометь: `требует данных о структуре сайта`

---

## Секция 13: Schema

**Когда:** после контент-плана.

Выбери по архетипу:
- `how_to_guide` → `HowTo` (если есть шаги)
- FAQ в контенте → `FAQPage`
- `comparison_page` → `ItemList`
- Всегда: `Article`

**Правило:** schema описывает только ВИДИМЫЙ контент страницы. Нельзя добавлять HowTo schema без реальных шагов.

---

## Секция 14: Редакционные правила

**Когда:** после всех контент-секций.

Адаптируй под:
- YMYL-уровень (строже = больше ограничений)
- Аудиторию (эксперт vs обычный пользователь)
- Тон (формальный vs дружелюбный)

Обязательно для YMYL: дисклеймер, разделение общих правил и ситуационных.

---

## Секция 15: Чек-лист приёмки

**Когда:** последним.

Пройди по каждому пункту. Если хотя бы один не выполнен — бриф не готов.

---

## Типичные ошибки Stage 2

| Ошибка | Как избежать |
|---|---|
| Выдуманные факты (суммы, сроки) | Пиши `TBD` в Stage 2, проверь в Stage 3 |
| «Рассказать о теме» вместо задачи | Формулировать «пользователь должен...» |
| Один outline для всех запросов | Следовать архетипу из Stage 1 |
| Триплеты без типов | Указывать baseline/enrichment/risk/excluded |
| FAQ без ответов | Каждый вопрос = полный ответ с источником |
| Конкуренты без решений | Таблица «сигнал → решение» |
| Нет сценарных развилок | Проверить: зависит ли intent от ситуации? |
| Фабрикованные URL | Использовать base-URL или verified URL |
| Нет образца документа | Проверить: предполагает ли тема составление документа? |
